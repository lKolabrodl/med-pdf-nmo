# ADR 001: Архитектура predictor для масштабирования на новые PDF

## Статус

Принято.

## Контекст

Predictor должен выбирать один или несколько правильных вариантов ответа по PDF медицинских рекомендаций НМО. Runtime не использует LLM, transformer inference, внешние интеллектуальные сервисы и не читает answer key во время inference.

Корпус будет расти от десятков PDF к тысячам PDF. Поэтому улучшения должны опираться на общие свойства документов: текст, нормализацию, структуру рекомендаций, строки, таблицы, списки, числовые условия и OCR-искажения. Нельзя кодировать конкретные вопросы, файлы, страницы или медицинские факты из текущего eval-набора.

## Решение

Predictor остается ансамблем небольших scorer-модулей. Каждый scorer отвечает за один тип evidence:

- lexical/search scorer'ы ищут обычные текстовые совпадения;
- focused/line scorer'ы проверяют локальные окна и строки рядом с фокусом вопроса;
- recommendation scorer'ы связывают вариант ответа с блоком клинической рекомендации;
- coordinate-table scorer'ы используют PDF item coordinates, чтобы восстанавливать ячейки, строки и группы таблиц;
- biomedical-symbol scorer'ы нормализуют латиницу, гены и OCR-подмены;
- selection слой калибрует raw score и выбирает single/multi набор.

Новый scorer должен иметь явный gate: он включается только для вопроса и структуры PDF, где его предположение оправдано. Это важнее, чем локальный прирост на одном примере.

## Почему так

Плоские совпадения по всему PDF быстро достигают потолка: варианты ответов часто встречаются дословно в нерелевантных разделах, комментариях, соседних рекомендациях и таблицах. Поэтому качество должно расти через структурное evidence: принадлежность к одной строке таблицы, одному списку, одному блоку рекомендации или одному предложению.

Такой подход масштабируется лучше, чем подгонка порогов:

- scorer можно валидировать отдельно на dev/holdout;
- новые PDF добавляют новые форматы, а не новые hardcode-правила;
- ошибки проще классифицировать по источнику evidence;
- риск leakage ниже, потому что inference видит только PDF, вопрос и варианты.

## Правила развития

1. Рефактор без изменения поведения должен проходить `npm test`, `npm run typecheck`, `npm run eval` и `npm run eval:holdout`.
2. Изменение логики predictor всегда сопровождается eval и записью в `docs/iteration-log.md`.
3. Если dev растет, а holdout падает, изменение считается подозрительным и должно быть отклонено или сужено.
4. Словари допустимы только маленькие и предметно-нейтральные: OCR-подмены, общеупотребимые медицинские сокращения, единицы измерения, пути введения.
5. Для multi-вопросов приоритет у структурных групп, а не у отдельных независимых score.

## Масштабирование

Ближайший путь масштабирования:

- развивать `coordinate-table` до полноценной модели таблиц: caption, headers, rows, continuations, column groups;
- выделить list/table membership extraction для случаев, где весь список или таблица отвечает на вопрос;
- сегментировать clinical recommendation blocks, чтобы не смешивать рекомендацию, комментарий и соседний пункт;
- добавлять diagnostics по классам ошибок, чтобы новые итерации выбирались по общим паттернам;
- пробовать маленький non-LLM calibrator только после появления богатых структурных признаков и только с group split по PDF.

## Последствия

`predictor.ts` должен постепенно становиться orchestration-файлом: подготовка runtime-контекста, вызов scorer'ов, агрегация score, selection и публичный API. Большие предметные блоки должны жить в `src/predictor/scorers/*`, а общие текстовые утилиты в `src/predictor/text-utils.ts`.
